# Архітектура вебсистеми перевірки текстів «Незбіг»

## 1. Призначення системи

«Незбіг» — вебзастосунок для попередньої перевірки текстових документів за трьома напрямами:

1. пошук текстових збігів у відкритих інтернет-джерелах;
2. локальна оцінка стилістичних ознак машинно згенерованого тексту;
3. редактор стилю, який прибирає шаблонні формули без навмисної зміни фактів, модальності та авторської пунктуації.

Система формує пояснюваний звіт: показує знайдені джерела, метрики збігу, локальні AI-ознаки, окрему думку зовнішньої мовної моделі та технічні примітки обробки документа.

Сервіс не є юридичним доказом плагіату або машинного авторства. Його результат — індикатор ризику, який потребує перевірки викладачем, редактором або автором роботи.

    REPORT --> UI

## 4. Потік обробки документа

### 4.1. Вхідні формати

Застосунок приймає вставлений текст і файли TXT, Markdown, CSV, JSON, RTF, DOCX та PDF. Розмір файлу на сервері обмежено 50 MB.

Для DOCX і вставлення з Word одночасно підтримуються два синхронізовані представлення:

- plain text зі збереженими межами абзаців для пошуку та AI-аналізу;
- очищений HTML для форматованого preview, редактора стилю та копіювання назад у Word.

Форматований preview не використовується у формулах збігу: HTML-теги, шрифти й таблиці не повинні впливати на текстовий скоринг.

HTML із буфера Word проходить allow-list очищення. Прості правила класів Word перетворюються на дозволені інлайн-стилі. Безпечні класи `Mso*`/`WordSection*` і властивості `mso-*` зберігаються окремо для коректного round-trip у Word: вони несуть нумерацію списків, рівні структури та правила пагінації, хоча браузер не використовує їх для звичайного відображення. Залишаються вирівнювання, розмір і сімейство шрифту, накреслення, відступи, закладки, посилання на виноски та оформлення таблиць. Скрипти, форми, iframe, виконувані CSS-вирази й небезпечні URL не переносяться.

Після редактора стилю сервер не генерує HTML заново з plain text. Спільний модуль `textAlignment` вирівнює початкові та змінені абзаци, а потім розподіляє змінені токени по початкових текстових вузлах. `formattedDocument` застосовує це до HTML, а `formattedDocx` — безпосередньо до `word/document.xml` всередині вихідного OOXML-пакета. Завдяки цьому замінене слово успадковує формат свого початкового фрагмента, а інші частини DOCX, зокрема стилі, таблиці, зображення, колонтитули та службові зв'язки, не перебудовуються. Клієнт використовує `ClipboardItem` одночасно з MIME-типами `text/html` і `text/plain`, тому Word отримує форматовану та резервну текстову версії.

### 4.2. Попередня обробка

Модуль `documentPreprocess.ts` виконує:

- відсікання програмного коду через `proseFilter.ts`;
- автоматичний пошук титульної або службової частини курсової роботи;
- початок перевірки з маркерів «ЗМІСТ», «ВСТУП», «РОЗДІЛ 1» або `Introduction`;
- нормалізацію пробілів для алгоритмів зіставлення.

Заголовки «ВСТУП», «РОЗДІЛ», «ВИСНОВКИ» самі по собі не є AI-ознаками.

## 5. Перевірка на текстові збіги

### 5.1. Повне покриття

Документ ділиться на перекривні фрагменти. Для великих текстів розмір фрагмента збільшується, але кількість фрагментів обчислюється так, щоб кінець документа також був перевірений. Типовий overlap становить близько 18% розміру фрагмента.

Формула кількості фрагментів:

```text
step = chunkWords - overlapWords
chunks = ceil((wordCount - overlapWords) / step)
```

Це усуває ситуацію, коли для довгого документа аналізуються лише перші N частин.

### 5.2. Побудова пошукових запитів

Для кожного фрагмента система:

1. розглядає фрази з різних чвертей фрагмента;
2. оцінює їх за довжиною слів, кількістю рідкісних довгих токенів, цифр і лексичною унікальністю;
3. формує кілька точних запитів у лапках;
4. додає запит із найбільш інформативних ключових слів;
5. у глибокому режимі збільшує кількість географічно розподілених фраз.

Такий підхід знижує залежність від першого речення й підвищує шанс знайти джерело, якщо збіг розташований у середині або наприкінці фрагмента.

### 5.3. Пошукові провайдери

- DuckDuckGo працює без API-ключа й використовується як базовий відкритий індекс.
- Google Custom Search використовується за наявності `GOOGLE_SEARCH_API_KEY` і `GOOGLE_SEARCH_ENGINE_ID`.
- Brave Search використовується за наявності `BRAVE_SEARCH_API_KEY` як незалежний вебіндекс. Провайдер має щомісячні безкоштовні кредити, але потребує окремої реєстрації.
- Semantic Scholar підключається у глибокому режимі для академічних статей та анотацій.
- OpenAlex підключається у глибокому режимі за наявності `OPENALEX_API_KEY`. Система відновлює анотацію з `abstract_inverted_index` і використовує DOI або open-access landing page як адресу джерела.

Результати різних провайдерів об'єднуються й дедуплікуються за канонічною URL. Пошуковий snippet є лише підказкою для hydration: він відображається у звіті, але не впливає на загальний відсоток плагіату, доки сервер не прочитає саму сторінку й не підтвердить спільний уривок.

Результати індексів змішуються циклічно, тому один провайдер не витісняє всі інші лише через порядок виконання запитів. Окрема черга допускає не більше трьох одночасних запитів до одного провайдера. Після трьох послідовних помилок circuit breaker призупиняє конкретний провайдер на 60 секунд ще до запуску задач, що чекали в черзі. Решта індексів продовжують працювати, а звіт окремо показує кількість успішних запитів, помилок і timeout.

### 5.4. Hydration вебсторінок

Для перспективних кандидатів сервер завантажує HTML, видаляє `script`, `style`, навігацію, форми й службові блоки, після чого виділяє текст `article`, `main` або `body`.

Через обмеження часу serverless-функцій для довгих документів застосовується двофазний режим:

1. швидкий пошук кандидатів для всіх фрагментів;
2. дочитування повних сторінок лише для найсильніших збігів.

### 5.5. Формула збігу

Для кожної пари «фрагмент — сторінка» обчислюються:

- `overlapPercent` — частка слів фрагмента, наявних у джерелі;
- 3-gram і 5-gram containment;
- `longestRun` — найдовша безперервна спільна послідовність слів;
- `fullTextRank` — частотне ранжування через локальний full-text index;
- `winnowing fingerprints` — стабільні хеш-відбитки 5-грам.

Поточна зважена формула:

```text
score = 100 × pageBonus × (
  0.10 × tokenOverlap +
  0.30 × phraseOverlap +
  0.24 × longestRunScore +
  0.26 × winnowingOverlap +
  0.10 × fullTextRank
)
```

`longestRun` обчислюється розрідженим динамічним алгоритмом за позиціями однакових токенів. Він не перебирає повну матрицю «кожне слово × кожне слово» і повертає координати збігу. Завдяки цьому звіт показує точний спільний уривок у поданому тексті та на підтвердженій сторінці.

Підсумковий відсоток агрегує до восьми найсильніших збігів із `confidence = page`. Результати з `confidence = snippet` залишаються пошуковими підказками й мають нульову вагу в підсумку. Це захищає від ситуації, коли пошуковик просто повторив точну фразу запиту у snippet, а текст сторінки недоступний або не містить цієї фрази.

### 5.6. Winnowing

Winnowing обирає правий мінімальний хеш у кожному ковзному вікні хешів k-грам. На відміну від попереднього агрегування хешів у дерево, fingerprints не залежать від позиції абзацу в документі. Додавання стороннього тексту перед скопійованим уривком не руйнує його відбиток.

Для параметрів `k` і `w` алгоритм гарантує вибір щонайменше одного fingerprint для спільного підрядка довжиною не менш як `w + k - 1` токенів. Метод базується на роботі Schleimer, Wilkerson і Aiken, *Winnowing: Local Algorithms for Document Fingerprinting*.

## 6. Локальний AI-аналіз

### 6.1. Межі задачі

Локальний модуль працює без зовнішньої LLM і повертає евристичний індикатор ризику, а не доказ авторства та не калібровану статистичну ймовірність. Це принципове обмеження: результати сучасних детекторів залежать від мови, домену, довжини, моделі-генератора й подальшого редагування. Тому числовий бал завжди супроводжується вердиктом, надійністю, мовним покриттям і конкретними фрагментами для ручної перевірки.

### 6.2. Підготовка авторського тексту

Перед стилометрією система створює окреме представлення документа. З нього вилучаються:

- fenced code blocks, окремі високовпевнені рядки коду та типові inline-конструкції мов програмування;
- довгі прямі цитати, які характеризують автора джерела, а не автора документа;
- бібліографічний хвіст після пізнього заголовка «СПИСОК ВИКОРИСТАНИХ ДЖЕРЕЛ», `REFERENCES` або `BIBLIOGRAPHY`;
- службова титульна частина курсової роботи, яку окремо розпізнає загальний препроцесор.

Кількість вилучених слів зберігається у `aiExclusions` і показується у звіті. Академічні заголовки «ВСТУП», «РОЗДІЛ», «ВИСНОВКИ» не вважаються AI-ознакою й не беруть участі в оцінці ритму речень. Подвійний дефіс і тире також не є AI-маркером.

### 6.3. Групи ознак

Ознаки поділені на три незалежні канали, щоб один частий, але слабкий шаблон не визначав увесь результат:

1. `statistical`: коефіцієнт варіації довжини речень, локальна лексична різноманітність MATTR та повтори 4-грам;
2. `pattern`: формальні переходи, hedging, безособовий академічний стиль, повторювані українські й англійські LLM-конструкції, prompt-leak;
3. `structure`: повтори початків речень, симетричні конструкції, надмірно регулярні переліки та пунктуаційний профіль.

MATTR обчислюється ковзними вікнами по 50 токенів, тому довгий документ не отримує штучно низьку лексичну різноманітність лише через свій обсяг. Для кожного каналу беруться до трьох найсильніших ознак із спадними вагами `0.55 / 0.30 / 0.15`. Канали агрегуються як `0.38 × statistical + 0.42 × pattern + 0.20 × structure`; множник corroboration зростає лише тоді, коли активні щонайменше дві незалежні групи.

Окремі слова на кшталт «важливий» або «ефективний» мають малу вагу і враховуються лише у фразовому контексті. Цитати, числа, перша особа та академічна структура є запобіжниками від false positive: вони зменшують упевненість, але не можуть механічно обнулити кілька узгоджених AI-сигналів.

### 6.4. Повне сегментне покриття

Увесь підготовлений текст ділиться на перекривні вікна по 220 слів з overlap 55 слів. Останнє вікно завжди прив'язане до кінця документа, тому хвіст не пропускається. Для кожного вікна зберігаються 1-based координати слів, локальний бал, уривок і найсильніші докази.

Підсумок довгого документа поєднує:

```text
36% — оцінка всього підготовленого документа
14% — медіана сегментів
20% — верхній квартиль сегментів
22% — середнє трьох найсильніших сегментів
 8% — частка сегментів із вираженими ознаками
```

Локальний floor не дає великій людській частині приховати невеликий машинно написаний фрагмент. Водночас результат із сильним розкидом не оголошується «AI-текстом»: він отримує вердикт `mixed`, а користувач бачить підозрілі ділянки окремо.

### 6.5. Вердикт, мова й надійність

`aiVerdict` має шість значень:

- `insufficient`: після вилучень залишилося менше 80 слів; у головній картці показується «—», а не хибні `0%`;
- `low`: достатній узгоджений текст без вираженого кластера ознак;
- `uncertain`: низька надійність, обмежене мовне покриття або слабкі суперечливі сигнали;
- `mixed`: у документі є локальні підозрілі й локальні непідозрілі сегменти;
- `elevated`: підвищений узгоджений ризик;
- `high`: сильний узгоджений ризик у кількох каналах.

Мовний модуль окремо визначає українську, англійську, змішану або непідтримувану вибірку. Для російської та інших непідтримуваних мов числовий результат не видається за впевнений висновок: вердикт стає `uncertain`, а надійність обмежується. Надійність `0..100` враховує обсяг, кількість сегментів, їхню узгодженість, кількість сильних незалежних ознак, мовне покриття та вилучений неавторський матеріал.

### 6.6. Окрема AI-думка

Після локального звіту клієнт асинхронно запитує зовнішню LLM через OpenRouter, а за невдачі через NVIDIA NIM. Timeout, rate limit, порожня відповідь або відмова всіх моделей не змінюють `aiProbability`, `aiVerdict` чи локальні сигнали. У UI це дві різні метрики: «AI-аналіз» та «AI-думка».

### 6.7. Наукові обмеження

Евристичний ансамбль корисний для triage, але не може встановити походження тексту. RAID і MULTITuDE демонструють нестабільність детекторів між доменами, моделями й мовами, а paraphrase attacks істотно знижують recall. Liang et al. виявили підвищений false positive для неносіїв англійської в окремій вибірці детекторів; новіша багатомовна переоцінка Mukherjee et al. не знайшла універсального систематичного bias у всіх сімействах. Практичний висновок для Незбігу: мовну валідність не можна припускати, її треба вимірювати на цільовому корпусі. Слабка ознака не перетворюється на категоричний висновок, а AI-бал не використовується як санкція.

Для наукової калібровки потрібен окремий український корпус із людськими та машинними текстами, розділений за автором, жанром, моделлю і способом редагування. Оцінювати слід ROC-AUC, macro-F1, expected calibration error та TPR при заздалегідь обмеженому FPR. До появи такого корпусу індикатор слід читати лише разом із вердиктом, надійністю та виділеними сегментами.

## 7. Редактор стилю

Редактор працює детерміновано й пояснює кожну групу змін. Він:

- видаляє чат-артефакти й рекламні формули;
- скорочує службові академічні заготовки;
- прибирає точні дублікати довгих речень лише всередині одного абзацу;
- переписує лише сталі канцелярні конструкції, а не окремі наукові або технічні терміни;
- зберігає абзаци, модальні слова «може/можуть», тире та типографічні лапки;
- не змінює код, оскільки код відсікається до текстового редагування файлу.

Нечіткі атрибуції на кшталт «дослідження показують» не замінюються вигаданим нейтральним джерелом. Редактор залишає твердження без зміни та додає примітку, що автору треба вказати конкретну працю. Однакові рядки в різних абзацах або елементах документа також зберігаються, бо вони можуть бути коректними назвами таблиць, позицій чи повторюваних форм.

Редактор не позиціонується як засіб обходу AI-детекторів. Його мета — читабельність і зменшення шаблонності без навмисної підміни змісту.

## 8. Кешування й продуктивність

`MemoryTtlCache` кешує:

- пошукові відповіді на 30 хвилин;
- прочитаний текст сторінок на 60 хвилин.

Негативний результат hydration також кешується: якщо сторінка заблокована, має непридатний content type або не відповідає, один інстанс не повторює той самий повільний запит для кожного фрагмента. Кеш відрізняє відсутній ключ від збереженого `undefined`.

Кеш має обмеження кількості записів і придатний для одного serverless instance. Він не є глобальним між усіма інстансами Vercel. Для production-навантаження доцільно підключити Redis/KV з хешем нормалізованого запиту як ключем.

Пошукові задачі виконуються з обмеженою конкурентністю. Для дуже довгих документів збільшується chunk size, зменшується кількість результатів на фрагмент і застосовується двофазна hydration. Це компроміс між повним покриттям і 300-секундним лімітом функції.

У `vercel.json` для API-функції явно встановлено `maxDuration: 300`. Це відповідає максимальній тривалості Hobby-плану з Fluid Compute. Circuit breaker, позитивний і негативний кеші скорочують повторне очікування недоступних зовнішніх сервісів, але кожний фрагмент документа все одно проходить пошуковий етап.

## 9. REST API

| Метод і маршрут | Вхід | Результат |
| --- | --- | --- |
| `GET /api/health` | — | стан сервера |
| `POST /api/extract` | multipart file | plain text, DOCX HTML preview, file evidence |
| `POST /api/scan` | JSON text + settings | базовий локальний звіт |
| `POST /api/scan-file` | multipart file + settings | файловий базовий звіт |
| `POST /api/ai-opinion` | text + local signals | незалежна LLM-думка |
| `POST /api/ai-opinion-file` | file + local signals | файлова LLM-думка |
| `POST /api/humanize` | JSON text + optional sanitized HTML | відредагований текст, форматований HTML і журнал змін |
| `POST /api/humanize-file` | multipart file | відредагований текст та HTML зі структурою DOCX |
| `POST /api/export-docx` | original DOCX + revised text | відредагований OOXML-файл зі збереженим пакетом Word |

### Збереження форматування Word

Для DOCX сервер формує два незалежні представлення документа. Нормалізований plain text надходить до пошуку й AI-аналізу, а HTML-представлення використовується лише для preview, редагування, копіювання та експорту. Під час конвертації зберігаються заголовки, абзаци, порожні рядки, списки, таблиці, зображення, жирний/курсивний/підкреслений текст, шрифт і кегль прямих run-стилів, вирівнювання, відступи, виділення та розриви сторінок. Перед показом HTML проходить allowlist-санітизацію.

Під час вставлення з Word клієнт спочатку очищує `text/html`, вставляє дозволену розмітку в поточну позицію курсора й одразу синхронізує форматований DOM із plain-text представленням для алгоритмів. У зворотному напрямку клієнт записує `text/html` і `text/plain` через сучасний Clipboard API; резервний шлях копіює тимчасове форматоване виділення. Для вставленого тексту доступний Word-сумісний `.doc` у UTF-8. Для завантаженого DOCX окремий endpoint змінює лише текстові вузли основної OOXML-частини й повертає справжній `.docx`, тому пакетні стилі, медіа та структура оригіналу зберігаються значно точніше, ніж після HTML-конвертації. Оформлення не впливає на відсотки перевірки.

## 10. Безпека та приватність

- HTML із Word проходить allow-list sanitizer до показу, повторного вставлення та копіювання.
- Сервер приймає лише відомі типи файлів і має ліміт розміру.
- Для завантаження джерел дозволені тільки HTTP/HTTPS URL.
- Скрипти, iframe, форми й навігаційні блоки видаляються з HTML сторінки.
- API-ключі зберігаються лише у змінних середовища й не потрапляють у клієнтську збірку.
- Файли обробляються в пам'яті та не записуються на постійний диск.

Зовнішнім пошуковим системам надсилаються текстові фрази з документа, а LLM-провайдерам — текст для додаткової оцінки. Це потрібно явно зазначити в політиці приватності перед використанням із конфіденційними роботами.

## 13. Обмеження та напрями розвитку

1. Жоден безкоштовний пошуковий API не надає повної копії інтернету. Закриті репозитарії, платні журнали, внутрішні дипломні бази та сторінки без індексації залишаються недоступними.
2. Для справжнього університетського антиплагіату потрібен власний ліцензований корпус робіт і попередньо побудований інвертований індекс fingerprints.
3. Локальний AI-скоринг треба калібрувати на українському контрольному корпусі та показувати довірчий інтервал.
4. Для пошуку перефразувань доцільно додати мультимовні embeddings і reranker, але тільки як другий етап після точного lexical matching.
5. Для великих документів production-архітектура має використовувати чергу задач, постійний кеш і polling/SSE замість однієї довгої serverless-відповіді.
6. PNG-звіт варто доповнити PDF/A-експортом і підписом версії алгоритму.
7. DOCX preview зберігає семантичне форматування, але Mammoth не відтворює точну пагінацію, колонтитули, плаваючі фігури та складні текстові блоки Word піксель у піксель. Завантажений після редагування DOCX зберігає ці OOXML-частини оригіналу, однак текст усередині колонтитулів, фігур, приміток і виносок редактор стилю поки не змінює.
